# ToolLtm 設計書 (v0.8 / 2026-07-07)

バーライフタイム台帳 + 加工物記録。単一HTML(index.html) + localStorage。GitHub Pages配置済み（githubtaro.github.io/toolltm-app/）。ローカルfile://版と併用可（localStorageはオリジン単位で別管理 → 移行はJSONバックアップ/復元）。

## 1. 目的と設計原則
- CORiTEC 350i 4台（d1/d2/w1/w2）のバー消耗・交換・加工物別実消費を、機械が吐くファイルから自動集計する
- **決定的処理はJS、LLMに時刻演算をさせない**（チャット運用の破綻からの教訓）
- **記録が作業の副産物として溜まる構造**（入力コストがチャット以下でないツールは死ぬ）
- **ミスしても壊れない**: 全角/半角の正規化、ブロック材料のデフォルト仮定、取り違えガード、保存前プレビュー必須

## 2. データソース（機械側、読み取り専用）
- `C:\CNCWorkbench\Control\Toolchange\TchUni\ToolProperties.INI` … 正本。スロット別に Type(バー番号)/ActRunTime/MaxRunTime/ActCuttingTime/ErrStatusFlag/RefPosZTimeStamp
- 同 `MeasureToolRefPosZ.TXT` … 工具長測定ログ(2016年〜追記型)。装着=`Storing measuring result of tool N after toolchange`、返却=`Control measure of collet after depositing tool N`。requestedのみの行は装着と数えない
- 取得: USB直下の `collect_d1〜w2.bat`（マシン名ハードコード4本、画面に大文字表示で目視確認）→ `data\<machine>\<yyyymmdd_hhmm>\` に copy（更新日時保持）
- 取得条件: そのマシンがアイドルなら何時でも可。加工中は取らない
- **取り込み時のアプリ挙動**（USBフォルダ一括／手動貼り付け共通。毎回ここを忘れるので明文化）:
  - **検知日(detectedAt) = INIファイルの更新日時(mtime)**。バッチが更新日時保持でコピー → mtime ≒ その機械の最終加工時刻。手動貼り付け時のみ日時フィールドの手入力値。**検知日 ≠ 交換日**（機械は交換時刻を持たない。実交換は「前回観測〜検知日」の区間としか言えない）
  - **マシン判定はフォルダ名**（`data\<machine>\…` の d1/d2/w1/w2）。ファイル中身もcomputername.txtも見ない → マシンを決めるのはバッチが作るフォルダ名だけ。正しい機械で正しいcollectバッチを回すのが前提。前回比でバー構成50%未満一致だと取り違え確認ダイアログ
  - **重複スキップ = 同一マシンでmtime±2分以内**。同じ`data`フォルダを何度投入しても二重登録なし（USB原本も消さない）
  - **処理順 = mtime昇順**（差分の連鎖を正しく繋ぐため）。複数日ぶんまとめて投入可
  - 交換検知は手動貼り付けと同一（buildDiff→commitSnapshot）が取り込み時に自動実行 → **通常運用で手動貼り付けは不要**（手動はテキストしか無い時の非常口）

## 3. 確定した機械仕様（実測で検証済み）
- ActRunTime: 交換操作で0リセット。空転も加算される（切削と区別なし）
- ActCuttingTime: **交換を跨いで永続累積**（純切削時間）。交換時差分=バー1本の生涯切削時間
- ErrStatusFlag: 256=MaxRunTime超過。交換や再起動では消えず、**そのバーの次回使用時に再評価されて消える**。1040=空スロット/未装着 → パース除外
- RefPosZTimeStamp: 測定のたび更新（交換専用の印ではない）→ 交換検知には使わない
- 要交換判定はアプリが `ActRunTime >= MaxRunTime` を自前計算（flagは参考バッジのみ）
- 交換検知 = 前回スナップショット比でRunTimeが減少（唯一の主判定）
- 日付表記: 制御PCはドイツ式 DD.MM.YYYY

## 4. データモデル（localStorage key: `toolltm.v1`）
```
{ snapshots:[{id,machine,takenAt,tools:[{slot,type,run,max,cut,flag,refTs}]}],
  exchanges:[{id,machine,type,detectedAt,prevTakenAt,finalRun,cutAtExchange}],
  segments:[{machine,slot,a,b,broken?}],
  jobs:[{id,date,machine,hhmm,material,thick,blank,slot,isBlock,kind,template,
         templateName,teeth,units,abut,pontic,veneer,userdef,note,
         expectedBars,materialAssumed?,needUnitInput?}] }
```
- パース対象工具: Type≠0 かつ flag≠1040
- スロット→バー解決: 「セグメント以降で最初のスナップショット」から。**スロット主キー**（w2のINI Type誤記録事件のため、Type値を無条件に信用しない）
- 12時間超のセグメントは「跨ぎ（電源断/装着放置）」として合計から除外

## 5. 加工物記録パーサー（旧・引き継ぎ定義ファイルのJS実装）
- 実装済みルール: C(NC名パース)・I(略語辞書)・A(バー割当)・E(ユニット補完)
- 削除: D(AI採番) / 実装済みにつき不要: F(差分計算) / 保留: B(insurance制約)・G(3Dプリンタ)・H(フェイルセーフ)
- NC名: `日付_素材_厚さ_(シェード)_マシン_ブランク_日付_HHMM.iso`。シェード抽出は未実装(Phase 2)
- ブロック判定: ファイル名にアルファベット無し。**材料はCAD/CAM冠と仮定**(実態99%、青タグ表示)。注記の`CAD`で確定、`PEEK`で上書き
  - 第3信号(暗黙知・未実装): CADブリッジをブロックで削る時のみテンプレが`Coping Bridge Fiber Block`になる。テンプレ名に`Fiber`を含む=ブロック確定として使える（通常のCAD Brはディスクで`Coping Bridge`）
- ブロックサイズは注記の箱表記トークン(14/14L/15XLL)から。NC名からは物理的に取得不能
- テンプレート名(insurance_v2等)は文字列保持のみ、辞書なし
- 注記の全角数字/スペース/＃は正規化して吸収
- 注記探索は次の.iso行で打ち切り（隣のジョブの注記を拾わない）
- セグメント突合: ジョブのHHMM以降、期待バー一巡まで（同一バー再登場か期待外バーで打ち切り）。段取り時間は含まない
- **他者ジョブ**: 自分のPCのhyperDENTログだけを貼る運用で構造的に除外（ログはPC単位）

## 6. 運用ルール
1. 平常: アイドル時にバッチ→帰宅後 `data` フォルダごと取り込み（重複は自動スキップ、USBの原本は消さない）
2. 交換はINI 2枚でサンドイッチ: **直前1枚**（finalRunがほぼ正確値になる。取らないと前回取得時点の下限値）＋**直後1枚**（即検知。同一間隔内の複数交換が1件に潰れるのを防ぐ）。スロット構成変更日も同様。※同一台で複数本まとめて交換する場合は台ごとに直前・直後の2枚で足りる（検知はType単位で独立）。潰れるのは「同じ番号のバーを1枚も挟まず2回交換」した時のみ
3. 手動貼り付けは非常口（チャットのテキストしか無い時のみ）。取得日時の手入力を誤ると偽交換検知が出る
4. 保存前にプレビューを目視（交換検知・要交換・仮定タグの確認）
5. localStorageはJSONバックアップを定期的に（移行・復旧用）
6. **正本はGitHub Pages版の一本（v0.8以降）**。file://版は凍結・新規入力禁止。復元（JSONから復元）は**全上書き・マージなし**のため、**古いJSON→Pagesへの復元は厳禁**（Pages側の新データが全消しになる）

## 7. 既知の制限 / Phase 2候補
- 同一バーを複数回使う複雑加工順では突合が打ち切られる（精緻化候補）
- シェード抽出未実装 / ルールB(w1×insurance_v2)バリデーション未実装
- 全台横断サマリ画面なし（差分ビューは登録時プレビューのみ）
- 手動貼り付けの時系列逆転ガードなし（「大半のバーが同時減少」警告が候補）
- 交換検知は**保存時に一度だけ**実行され、**取り込み順に依存**する。後から古いスナップショットを追加しても既存スナップショットは再差分されない（例: 初回取り込みで欠けた過去INIを後入れしても交換行は復活しない。復活させるには後続スナップショットを削除して再取り込み）
- localStorage 5MB上限。数年分のセグメントで要間引き or Supabase移行
- 異常データ（空回り・再製等）の除外フラグ未実装（旧ルールF後半）
  - 再製時の非対称性: バー消耗は両ジョブとも実消費（二重計上ではない）。二重計上になるのはjobs側の歯数/ユニット集計のみ → 除外フラグはunits集計にだけ効かせること
  - **自動分離は不可能**: ディスク未装着の空回りでも機械はディスク有無を検知できず切削送り中はCuttingTimeを加算する可能性が高い → 「RunTime増・Cut平坦」のようなクリーンな署名が出ない。したがって除外は数値検知ではなく**人手フラグ（セグメント/ジョブに手動チェック）に頼るしかない**
- 差分・交換検知の突合は**Type主キー**（スロット移動しても同一バーとして追従させる意図。セグメントのスロット主キーとは別物）。同一Type値が複数スロットに重複記録されると先勝ちで誤差分の可能性（現状のw2誤記録では重複なし・実害なし）
- **【実装済み v0.8・案A】Br内訳(支台/ポン)警告の廃止**: 内訳データに現状使い道なし（バー消耗分析は全ユニット数だけで足りる）。「あれば使えるかも」で入力コストを増やすのは設計原則「入力コストがチャット以下でないツールは死ぬ」に反する → **needUnitInput判定と「Br内訳要確認」バッジを削除**。歯番からの全数把握のみとする。
  - 保持する部分: 注記に支台/ポンが**書かれていれば**パーサーは引き続き拾って保存（機会的取得・警告なし）。将来使い道が出たら、歯番は保存済みなのでFDI隣接順マッピングで遡って導出可能
  - 却下した案B(歯番から支2/ポ1を仮定): FDI象限跨ぎで算術不能(#13-23を11と誤算)なりマッピング必須 → 使い道未定の今は投機的すぎる
